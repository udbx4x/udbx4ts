# 测试与兼容性规范

## 1. 目的

本规范用于约束测试设计和兼容性验证，保证 `udbx4ts` 的实现能够稳定复现 UDBX 行为，而不是只在本地样例上“看起来可用”。

## 2. 总体原则

测试优先级必须遵循：

1. 正确性
2. 兼容性
3. 跨运行时一致性
4. 性能

第一阶段不得用性能测试替代正确性和兼容性验证。

## 3. 测试分层规则

必须建立以下测试层：

- 单元测试
- 集成测试
- 浏览器运行时测试
- Electron 运行时测试
- `udbx4j` 对照测试

禁止：

- 只有单元测试没有运行时测试
- 只有示例手工验证没有自动化测试

## 4. 单元测试规则

以下模块必须有单元测试：

- `BinaryCursor`
- `BinaryWriter`
- GAIA Header
- Point/Line/Polygon codec
- SQL 结果映射
- 系统表元数据转换

要求：

- 测试输入明确
- 断言结构完整
- 对边界条件有覆盖

## 5. 兼容性测试规则

兼容性测试必须以 `udbx4j` 为基准。

必须覆盖：

- 编码字节与参考样例对照
- 解码结果与参考样例对照
- 新建库系统表结构对照
- 数据集元信息对照

允许：

- 在存在明确原因时记录已知差异

禁止：

- 在没有说明的情况下接受“结果差不多”
- 只做人工 eyeballing，不做自动化断言

## 6. 运行时测试规则

浏览器运行时测试必须覆盖：

- Worker 启动
- `sqlite3.wasm` 装载
- OPFS 模式
- fallback 内存模式
- 文件导入导出

Electron 运行时测试必须覆盖：

- 数据库打开
- 新建数据库
- IPC 调用链
- 事务写入
- 文件备份与关闭

## 7. 测试数据规则

测试夹具必须集中管理在 `tests/fixtures`。

必须包含：

- `SampleData.udbx`
- 关键 GAIA golden bytes
- 新建库预期结果样例

禁止：

- 把测试二进制散落在实现目录
- 在测试中内联过大的字节样例而不沉淀夹具

## 8. 回归规则

以下改动必须触发相关回归测试：

- `SqlDriver` 契约变化
- GAIA codec 改动
- 系统表结构改动
- 浏览器 Worker / OPFS 链路改动
- Electron IPC 与驱动改动

## 9. 缺陷处理规则

发现兼容性缺陷后，必须：

1. 先补失败测试
2. 再修复实现
3. 再记录原因和影响范围

禁止：

- 先改实现再补测试，导致问题不可复现
- 只在示例里绕过问题而不修复核心逻辑
