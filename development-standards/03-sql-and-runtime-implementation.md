# SQL 抽象与运行时实现规范

## 1. 目的

本规范用于约束 SQL 抽象、浏览器运行时和 Electron 运行时的实现方式，确保双平台闭环与核心层复用能力成立。

## 2. SQL 抽象规则

所有底层数据库访问必须经过统一的 `SqlDriver` / `SqlStatement` 契约。

必须：

- 对外保持异步接口
- 保持参数绑定能力
- 支持事务包装
- 支持语句生命周期管理

禁止：

- 在核心层直接调用平台私有 SQLite API
- 绕过抽象在不同模块里各自接数据库
- 让数据集类依赖具体驱动实现类型

## 3. SQL 使用规则

项目采用显式 SQL，而非 ORM。

必须：

- 让 SQL 出现在明确的仓储或数据集实现中
- 保持参数化查询
- 让结果映射逻辑集中管理

禁止：

- 引入 ORM 或复杂 DSL
- 拼接未转义的用户输入
- 在多个数据集中复制同样的 SQL 片段而不抽取公共能力

## 4. 浏览器运行时规则

浏览器运行时必须采用：

- `sqlite3.wasm`
- Dedicated Worker
- OPFS 首选模式
- 无 OPFS 时的 fallback 内存模式

必须：

- 所有数据库执行位于 Worker 内
- 主线程只通过 RPC 调用数据库能力
- 支持导入 `.udbx`
- 支持导出当前数据库

禁止：

- 在主线程直接持有 SQLite 连接
- 把 OPFS 当成唯一可用模式
- 在浏览器中默认直接原地修改用户文件

## 5. 浏览器文件策略

浏览器端必须采用“工作副本”模型。

标准流程：

1. 用户选择文件
2. 导入到工作区
3. Worker 中打开数据库
4. 执行读写
5. 用户显式导出

允许：

- 使用 File System Access API 优化体验
- 使用 Blob 下载作为 fallback

禁止：

- 把用户文件句柄能力当成前提
- 让导入、导出逻辑散落在多个模块中

## 6. Electron 运行时规则

Electron 端第一阶段采用成熟原生 SQLite 驱动。

必须：

- 数据库访问位于主进程或受控服务层
- 渲染进程通过 IPC 调用数据库能力
- 文件生命周期由运行时层统一管理

禁止：

- 在渲染进程直接创建数据库连接
- 让 UI 代码直接操作 SQL
- 在不同窗口中无序共享连接实例

## 7. 事务与资源管理规则

必须：

- 所有批量写入使用事务
- 所有 statement 都有明确关闭路径
- 所有连接都有明确关闭路径

禁止：

- 依赖 GC 释放数据库资源
- 在异常路径中遗漏 statement 或连接清理
- 在 Electron 中默认开启不受控的多连接写入

## 8. 运行时错误处理规则

必须区分以下错误类型：

- 数据库打开失败
- 文件导入导出失败
- Worker / IPC 通信失败
- SQLite 执行失败
- 能力不可用，如 OPFS 不可用

错误处理必须：

- 明确标记发生在哪个运行时
- 保留原始错误上下文
- 提供可上抛给上层的统一错误模型

## 9. 扩展控制规则

第一阶段不得新增与技术方案不一致的运行时扩展，例如：

- React Native SQLite 适配
- 服务端代理模式
- Web 端多 worker 分布式数据库方案

若确需扩展，必须先修改技术方案与执行计划，再开始实现。
